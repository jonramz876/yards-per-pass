## 2. Pipeline Changes (scripts/ingest.py)

### 2.1 New PBP Column Required

Add `cp` (completion probability) to `REQUIRED_PBP_COLS`. This is a standard nflverse column generated by the Ben Baldwin xgboost model. Available on pass attempts where a completion/incompletion outcome occurred. NULL on sacks, scrambles, spikes, throwaways, and plays where air_yards is NULL. ~5-10% of regular pass attempts have NULL cp even in modern seasons (screens, missing charting data). Quality degrades before 2010 but we start at 2020 so no concern.

### 2.2 New QB Stats (5 columns)

Add to `aggregate_qb_stats()`:

| Stat | Column Name | Formula | Notes |
|------|-------------|---------|-------|
| TD% | `td_pct` | `touchdowns / attempts * 100` | Already have `touchdowns` and `attempts` |
| INT% | `int_pct` | `interceptions / attempts * 100` | Already have both |
| SK% | `sack_pct` | `sacks / (attempts + sacks) * 100` | Standard PFR formula (sacks excluded from attempts) |
| SCR% | `scramble_pct` | `scrambles / dropbacks * 100` | `scrambles` already computed (line 315) but NOT exported — compute inline, don't add raw scrambles column |
| Total EPA | `total_epa` | `dropback_epa_sum` (already computed at line 311, just not exported) | Use existing `.dropna().sum()` to match mean() behavior |

**Fantasy points:** NOT added to pipeline. Already computed client-side via `qbFantasyPoints()` in `lib/stats/fantasy.ts` (4pt pass TD, -2 INT, -1 FL, rush_yds/10, 6 rush_TD). Keep existing scoring toggle (PPR/Half/Std) on Fantasy tab.

**Implementation:** All 5 are trivial arithmetic on existing aggregations. No new groupby needed. Compute scramble_pct inline from transient `scrambles` column (don't persist raw scrambles to DB).

### 2.3 New Receiver Stats (4 columns)

Add to `aggregate_receiver_stats()`:

| Stat | Column Name | Formula | Notes |
|------|-------------|---------|-------|
| AY% | `air_yards_share` | `player_air_yards / team_total_air_yards` | Same primary-team logic as target_share |
| CROE | `croe` | `catch_rate - mean(cp)` per receiver | Requires `cp` column from PBP |
| Recv SR% | `receiving_success_rate` | `mean(success)` on target plays | `success` column already in PBP |
| Total EPA | `total_receiving_epa` | `sum(epa)` on target plays | EPA already in PBP |

**CROE Implementation:**
```python
# Per receiver: expected_catch_rate = mean(cp) across all their targets
# CROE = actual catch_rate - expected_catch_rate
# NULL if <50% of targets have valid cp (avoids volatile estimates)
cp_by_receiver = target_plays.groupby('receiver_player_id').agg(
    expected_catch_rate=('cp', lambda x: x.dropna().mean() if x.notna().mean() >= 0.5 else float('nan')),
    cp_coverage=('cp', lambda x: x.notna().mean()),  # for logging
).reset_index().rename(columns={'receiver_player_id': 'player_id'})
rec = rec.merge(cp_by_receiver[['player_id', 'expected_catch_rate']], on='player_id', how='left')
rec['croe'] = rec['catch_rate'] - rec['expected_catch_rate']
# Log warning if any receivers have >20% missing cp
low_cp = cp_by_receiver[cp_by_receiver['cp_coverage'] < 0.8]
if not low_cp.empty:
    log.warning("%d receivers have >20%% targets missing cp", len(low_cp))
```

**Known bias:** CROE numerator (catch_rate) uses ALL targets, but expected catch rate uses only targets with valid cp. This is industry standard — same approach as most public CROE implementations. The 50% threshold guards against extreme cases.

**AY% Implementation:** Same pattern as `target_share` — compute `team_total_air_yards = target_plays.groupby('posteam')['air_yards'].apply(lambda x: x.dropna().sum())`, then divide player air yards by team total (primary team only for traded players). Use `.dropna().sum()` to handle NULL air_yards on throwaway plays.

## 4. Frontend: PFR Qualified Minimums

### 4.1 Default Thresholds

Replace arbitrary slider defaults with PFR-scaled minimums:

| Position | PFR Rule | Full-Season Default | Pro-Rated Formula |
|----------|----------|--------------------|--------------------|
| QB | 14 att/team game | 238 attempts | `14 * games_in_season` (scale to week) |
| RB | 6.25 att/team game | 106 carries | `6.25 * games_in_season` |
| WR/TE | 1.875 tgt/team game | 32 targets | `1.875 * games_in_season` |

**Note:** For receiver leaderboard, change the primary filter from `routes_run` to `targets`. PFR minimum is 32 targets. Remove the routes_run slider — YPRR/TPRR already account for route efficiency.

### 4.2 "Qualified" Toggle

Add a toggle next to the filter area: "PFR Qualified" (default: ON).
- ON: slider is **hidden** (not grayed out — avoids frustration). Show a badge instead: "PFR Qualified (238+ att)" / "(106+ car)" / "(32+ tgt)"
- OFF: slider appears and becomes manually adjustable (current behavior)
- When toggling ON after manual adjustment: snaps back to PFR minimum
- URL param: `?qualified=0` when OFF (default omitted)
- Add `qualified` to `buildParams()` managed-keys list (alongside tab, sort, dir, q, min)

### 4.3 Pro-Rating Logic

Determine current week from data: `max(week)` from latest season data. Scale threshold:
- `qualified_min = Math.round(pfr_per_game * current_week)`
- Example: Week 10 QB = `14 * 10 = 140 attempts`

## 5. Frontend: Position Rank Column

### 5.1 Dynamic Position Rank

Add "Rank" as first column in all leaderboards. Shows `QB1`, `QB3`, `WR25`, `TE4`, `RB7` etc.

**Calculation:**
- Among visible rows (after filtering by qualified threshold, position filter, search)
- Each tab defines a fixed `rankBy` stat (NOT whatever the user sorts by — prevents confusion where QB1 changes meaning per sort column)
- `rankBy` per tab: Overview=EPA/DB, Passing=yards, Rates=TD%, EPA=total_epa, Fantasy=fantasy_pts (QB). Similar per position.
- Position prefix from the player's position (QB for QB leaderboard, WR/TE/RB for receiver leaderboard based on actual position)
- When user sorts by a different column, the visual row order changes but position ranks stay fixed to the `rankBy` stat

### 5.2 Visual Style

- Bold, fixed-width column (60px)
- Gray background, monospace font
- Position abbreviation + number with no space: `QB1`, `WR25`

## 7. Frontend: Stat Group Sub-Tabs

### 7.1 Architecture: Tab Config Object Pattern

**Before touching any leaderboard file**, extract a shared config pattern to avoid 45+ top-level constants:

```typescript
// Per leaderboard: single config object replaces 6+ separate arrays/sets
const QB_TABS = {
  overview: { columns: [...], heatmapCols: new Set([...]), defaultSort: "epa_per_db", rankBy: "epa_per_db", defaultHeatmap: true },
  passing:  { columns: [...], heatmapCols: new Set([...]), defaultSort: "passing_yards", rankBy: "passing_yards", defaultHeatmap: false },
  rates:    { columns: [...], heatmapCols: new Set([...]), defaultSort: "td_pct", rankBy: "td_pct", defaultHeatmap: false },
  epa:      { columns: [...], heatmapCols: new Set([...]), defaultSort: "total_epa", rankBy: "total_epa", defaultHeatmap: false },
  fantasy:  { columns: [...], heatmapCols: new Set([...]), defaultSort: "fantasy_pts", rankBy: "fantasy_pts", defaultHeatmap: false },
} as const;
type QBTab = keyof typeof QB_TABS;
```

This replaces separate `ADVANCED_COLUMNS` + `STANDARD_COLUMNS` + `HEATMAP_COLS_ADVANCED` + `HEATMAP_COLS_STANDARD` + `VALID_ADVANCED_KEYS` + `VALID_STANDARD_KEYS` (6 exports per leaderboard) with one config object.

### 7.2 Shared Utility: formatStatValue()

Extract a single `formatStatValue(key: string, val: number): string` utility that both `formatVal()` and `formatAvg()` call. Avoids duplicating 12 new format cases × 3 leaderboards × 2 functions = 72 switch cases.

### 8.2 Percentile Handling for CROE

- Minimum 20 targets for CROE percentile pool inclusion (not routes — targets are more relevant for CROE)

- Higher CROE = better = higher percentile
- Range typically -15% to +15%

## 13. Risk Assessment

| Risk | Mitigation |
|------|-----------|
| `cp` column missing in older seasons | nflverse has `cp` from 2006+, quality degrades before 2010. We start at 2020. Fallback: set CROE to NULL if `cp` unavailable |
| `cp` NULL on subset of targets | 50% threshold: CROE = NULL if <50% targets have valid cp. Log warning if >20% missing |
| 5 tabs may overwhelm mobile | Horizontal scroll + scroll-snap + right-edge fade gradient hint |
| Position rank confusion (WR25 vs overall rank) | Label column clearly as "Pos Rank", tooltip explains |
| Pipeline runtime increase | Minimal — no new groupby operations, just arithmetic on existing aggregations |
| Breaking existing URL state | New tab names are different from `advanced`/`standard`. Old URLs fall back to `overview` (default) |
